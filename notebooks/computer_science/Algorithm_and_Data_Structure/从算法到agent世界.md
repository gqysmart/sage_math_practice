# 从算法到 Agent：以状态为中心的统一视角笔记

---

## 0. 当前理解的关键转折点

我逐渐意识到：

> **算法理论中最重要的不是“步骤”，  
> 而是“什么状态算解（solution state）”。**

而这，正是 Agent 把世界抽象为 **state / action** 的根本原因。

---

## 1. 算法的本质：状态保证，而不是过程描述

在理论意义上：

- 算法不是一串“聪明的步骤”
- 而是一个 **状态演化保证**

### 精确表述

给定：

- 状态空间 `S`
- 初始状态集合 `S_init`
- 解状态集合 `S_sol`
- 合法状态转移关系 `→`

算法正确性可以写成：

对所有执行路径 p，
如果 p 从 S_init 出发并终止于状态 s，
那么 s ∈ S_sol

> **路径可以不同，  
> 中间状态可以不同，  
> 但终止状态不能出错。**

---

## 2. 为什么“定义解状态”是第一性问题？

因为：

- 算法不因“过程优雅”而正确
- 只因“停在对的状态”而正确

这意味着：

- 正确性判定与时间无关
- 与执行顺序无关
- 与实现细节无关

👉 **这一步属于数学模型，而不是计算模型。**

---

## 3. 计算模型的角色：为状态演化提供语义

计算模型不定义“对或错”，它只限定：

- 什么是一步（时间）
- 状态如何表示
- 状态如何变化
- 执行路径是否区分

因此可以总结为：

> **计算模型定义“状态如何演化”，  
> 数学模型定义“哪些状态算成功”。**

---

## 4. 从算法自然过渡到 Agent

一旦我接受：

> **算法 = 对状态演化结果的保证**

那么以下抽象是不可避免的：

1. 世界必须被描述为 **状态（state）**
2. 行为必须被描述为 **动作（action）**
3. 决策的意义在于：  
   **把当前状态推向目标状态集合**

这正是 Agent 的基本形式。

---

## 5. Agent 的核心，不是“执行步骤”，而是“状态管理”

### Agent 世界观

- state：世界在某一时刻的完整描述
- action：对状态施加的可控改变
- trajectory：state → action → state → …

Agent 的“正确性”不在于：

- 执行了哪些步骤

而在于：

- 是否能把世界状态维持或引导进 **目标状态集合**

---

## 6. 为什么 Agent 强调“只看当前 state”？

一个成熟的 Agent 通常遵循：

> **决策只依赖当前状态，而不依赖历史路径**

这并不是工程偷懒，而是理论必然：

- 如果状态定义是完备的
- 那么历史已经被“折叠”进当前状态

这在 RL 中被称为 **Markov 性**，  
在算法理论中对应：

> **正确性不应依赖“怎么走到这里”。**

---

## 7. Agent = 算法在动态世界中的推广

传统算法隐含前提：

- 输入一次性给定
- 世界是静态的
- 算法结束即任务完成

Agent 的前提是：

- 世界持续变化
- 行为改变未来
- 没有“最终一次运行”

因此：

> **算法被“拉长”为一个持续运行的状态控制策略，  
> 这正是 Agent 的本质。**

---

## 8. Agent、RL 与离散结构的回归

即使：

- 状态是连续的
- 动作是连续的

Agent 的核心问题仍然是：

> 是否存在一条  
> `state → action → state → …`  
> 的轨迹，使状态进入并维持在目标集合？

这在结构上等价于：

- 图的可达性
- 博弈中的制胜策略
- 路径存在性问题

👉 **Agent 并没有远离算法理论，而是把它带进了开放世界。**

---

## 9. 一个现在对我来说非常清晰的对应关系

| 视角 | 关注点 |
|---|---|
| 数学模型 | 解状态集合 |
| 计算模型 | 状态 / 时间 / 路径 |
| 算法 | 全路径状态保证 |
| Agent | 持续的状态控制 |
| 启发式 | 无状态保证的经验策略 |

---

## 10. 当前阶段的一句话总结

> **当问题从“算一个答案”变成“持续与世界交互”，  
> 算法自然演化为以 state / action 为核心的 Agent；  
> 而这并不是 AI 的新发明，而是算法理论的必然延伸。**

---
