# M-001_Why_Math_Model_Disappears_In_System_Design.md
## 为什么在「业务 → 系统」建模中，数学模型会“退居幕后”？

### 0. 结论先行（Core Answer）
在业务到系统的建模过程中，数学模型并没有真正消失，  
而是被工程理性主动弱化、隐式化，并由「概念模型 + 设计模型」取代其中心地位。

这不是能力不足，而是目标函数发生了变化。

---

### 1. 两种建模路径的根本差异
你实际上面对的是两种完全不同的建模文明。

#### 1.1 科学 / 数学建模路径
现实问题
→ 抽象变量
→ 数学关系
→ 推导 / 证明
→ 算法

目标是：
- 真理性
- 可证明性
- 全局一致
- 最优 / 极值 / 收敛

数学模型在这里居于核心位置。

#### 1.2 业务 / 系统建模路径
业务需求
→ 业务模型
→ 概念模型
→ 设计模型
→ 可运行系统

目标是：
- 可运行
- 可演化
- 可维护
- 可协作
- 可控风险

系统建模追求的是**长期稳定运行**，而非形式化最优。

---

### 2. 数学模型为何不再适合作为中心？
#### 核心原因：业务问题不是“求解问题”，而是“协调问题”

数学模型最擅长处理的是：
- 连续量
- 可度量对象
- 稳定关系
- 全局规则
- 优化与极值

而业务系统面对的是：
- 人的行为与决策
- 法规与权限
- 例外与特判
- 状态变更
- 历史与责任归属

> 业务的本质不是函数关系，而是**规则、状态与责任的协调**。

这些内容无法自然表达为方程。

---

### 3. 谁取代了数学模型的位置？
#### 概念模型（Conceptual Model）

在系统工程中：
> 概念模型 = 被工程化、弱形式化的数学模型

它保留抽象能力，但放弃数学的严格性。

| 数学模型元素 | 概念模型中的对应 |
|---|---|
| 抽象对象 | Entity / Aggregate |
| 关系 | Association / Reference |
| 约束 | Business Rule / Invariant |
| 状态空间 | State Machine |
| 演化 | Event / Transition |

#### 代价与收益
放弃：
- 连续性
- 可推导性
- 解析解
- 全局最优

换来：
- 可实现
- 可维护
- 可扩展
- 可协作

---

### 4. 为什么系统会直接进入「设计模型」？
#### 因为“正确性”的定义发生了改变

数学中的“正确”：
- 推导成立
- 证明成立

系统中的“正确”：
- 在现实约束下长期稳定运行
- 不失控、不混乱、不崩溃

因此系统优先关注：
- 边界是否清晰
- 责任是否唯一
- 状态是否可控
- 故障是否可隔离
- 修改是否可局部化

这些目标通过**架构与设计模式**更直接实现，
而不是通过数学推导。

---

### 5. 数学真的“消失”了吗？
没有，它被埋进了三个位置。

#### 5.1 局部子系统中
一旦业务涉及：
- 定价
- 推荐
- 排队
- 路径规划
- 风控
- 调度

数学模型立刻回归，但只存在于**局部模块**，而非全系统。

#### 5.2 状态机与不变量中
- 状态转移 ≈ 离散动力系统
- 不变量 ≈ 约束条件
- 一致性 ≈ 局部守恒

只是用：
- UML
- DDD
- 代码约束

来表达，而不是数学符号。

#### 5.3 性能与运维模型中
系统上线后，数学以另一种形式出现：
- 队列论
- 概率分布
- 吞吐与延迟分析
- 容量规划

这是运行阶段的数学，而不是需求阶段的数学。

---

### 6. 本质判断：这是一次理性的取舍
可以这样总结：

| 数学建模 | 系统建模 |
|---|---|
| 全局一致 | 局部自治 |
| 形式严谨 | 工程可行 |
| 推导正确 | 运行稳定 |
| 静态结构 | 可演化结构 |

> 系统工程刻意牺牲了“形式化最优性”，  
> 换取“现实可控性”。

---

### 7. 判断边界（非常重要）
当问题的核心是：
- 优化
- 极值
- 收敛
- 概率
- 最优性

→ **数学模型必须居中**

当问题的核心是：
- 责任
- 权限
- 状态
- 演化
- 协作

→ **数学模型必须退居幕后**

### 9. 总结性陈述（可直接引用）
数学模型并没有在系统建模中消失，  
它只是从“中心舞台”退回到“局部引擎”。

当建模目标从“求解真理”转为“组织协作”，  
边界、主体与假定就自然取代了方程，  
成为系统模型的第一性要素。
